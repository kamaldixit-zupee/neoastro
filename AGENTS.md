# NeoAstro iOS – Agent Guide

Use this doc for any change inside this repository. It is the iOS user app for the NeoAstro astrology platform. The companion docs are [ARCHITECTURE.md](ARCHITECTURE.md) (deeper reference) and [CONTRIBUTING.md](CONTRIBUTING.md) (workflow).

## Repository at a glance

- **Single iOS target** generated from `project.yml` via XcodeGen.
- **Swift 5.10 + SwiftUI**, iOS 26.0 minimum, dark and light mode both, iPhone-portrait only.
- **No third-party dependencies** – stdlib + Apple frameworks only.
- **MVVM with `@Observable @MainActor`** ViewModels. No Combine, no `ObservableObject`.
- **Networking**: a single `URLSession` actor (`APIClient`) with auto-retry on 401 and three-shape envelope decoding.
- **Tokens** live in Keychain via `TokenStore`. Nothing else is persisted.

## Hard rules

1. **Do not hand-edit `NeoAstro.xcodeproj/project.pbxproj`.** It is regenerated by XcodeGen. Edit `project.yml` and run `xcodegen generate`.
2. **Do not introduce Combine.** Don't `import Combine`, don't add `@Published`, `ObservableObject`, `@StateObject`, or `@EnvironmentObject`. Use `@Observable @MainActor` + `async`/`await`.
3. **Do not bypass `APIClient`.** Every network call goes through `APIClient.shared.send`. It owns auth headers, device headers, refresh-on-401, and envelope detection. Adding a bare `URLSession` call breaks token refresh.
4. **Do not change `DeviceInfo.zupeeAppName` away from `com.neoastro.android`.** The Zupee gateway gates on package name per LOB. Changing it without a coordinated backend update will 403 every request.
5. **Do not strip envelope fall-throughs** in `APIClient.send`. The backend returns three shapes; all three must stay supported.
6. **Do not use `print` / `NSLog`.** Use `AppLog.<category>.info/error`. Add new categories to `AppLog.swift` rather than reusing the wrong one.
7. **Do not add caching layers** (Core Data, SwiftData, file caches) without a product requirement. The app is intentionally stateless beyond Keychain.
8. **Do not change the bundle ID** `varasol.MarathiCalendarPanchangam` casually – provisioning profiles and App Store Connect are tied to it.
9. **Treat `Models/API/`** as wire format. Do not add view-layer concerns (formatting, image caching) to those types; do that in the ViewModel.

## Adding a new feature (recipe)

1. Create `NeoAstro/Features/<Name>/` with `<Name>View.swift` and `<Name>ViewModel.swift`.
2. Add wire-format DTOs to `NeoAstro/Models/API/<Name>API.swift` (Codable, Identifiable/Hashable as needed).
3. Add an enum facade in `NeoAstro/Services/<Name>Service.swift` with `static func` calls into `APIClient.shared.send`.
4. Inside `<Name>ViewModel`: `@Observable @MainActor final class`, expose `async` methods, translate errors into `errorMessage: String?`.
5. Inside `<Name>View`: `@State private var vm = <Name>ViewModel()`. Wire navigation via `NavigationStack` + `.navigationDestination(item:)`.
6. If the feature is a tab, add the case to `AppTab` and the `Tab` entry in `MainTabView`.
7. Run `xcodegen generate` only if you added/removed source files outside the existing folder structure (XcodeGen globs handle most cases automatically – check `project.yml`).

## Adding a new endpoint (recipe)

1. Define request/response DTOs in `Models/API/`.
2. Add the call to the relevant `*Service`:
   ```swift
   static func foo() async throws -> FooResponse {
       try await APIClient.shared.send(
           Request(path: "/v1.0/foo", method: .GET),
           as: FooResponse.self
       )
   }
   ```
3. Inspect the actual envelope shape in stage – `APIClient.send` will pick the right one of the three, but if your DTO is wrong it will fall through all three and throw `decoding`. Log + adjust.
4. If the endpoint requires a non-standard header, pass it via `Request.extraHeaders`. Do not modify `APIClient.buildURLRequest`.

## When to ask before acting

- Touching `APIClient`, `TokenStore`, `DeviceInfo`, or the envelope types – these are load-bearing across every feature.
- Changing `Info.plist`, `project.yml` build settings, the bundle ID, or signing config.
- Adding a third-party dependency (SPM, CocoaPods, Carthage). The current "no deps" stance is intentional; first dep should be discussed.
- Adding analytics / crash reporting. There is none today; introducing one needs product/legal sign-off.
